# DPoP Security Architecture with Next.js & .NET 8 Web API

This project demonstrates a highly secure, modern authentication architecture using **DPoP (Demonstrating Proof-of-Possession at the Application Layer)** combined with the **Backend-For-Frontend (BFF)** pattern.

## Overview

Traditional OAuth 2.0 Bearer tokens are susceptible to theft. If an attacker intercepts a Bearer token (via network sniffing, XSS, or malware), they can replay that token to impersonate the user. 

**DPoP** solves this by cryptographically binding the access token to a specific client and a specific public/private key pair.

In this project, we implement a robust **Frontend -> BFF -> Backend** flow:
- **Frontend**: Next.js React Application
- **BFF (Next.js Server Actions)**: Node.js server handling the DPoP KeyPair and Token storage
- **Backend**: .NET 8 Web API enforcing DPoP validation

---

## Security Measures & Flow

### 1. The Backend-For-Frontend (BFF) Pattern
Storing sensitive tokens in the browser's `localStorage` or memory makes them highly vulnerable to Cross-Site Scripting (XSS) attacks. 

To mitigate this, the Next.js frontend employs a **BFF architecture**:
- **Zero Browser Exposure**: The React components (`page.tsx`) never touch, see, or handle the DPoP private key or the `.NET` Access Token. The browser only interacts with the Next.js server.
- **Secure Server State**: The DPoP ECDSA P-256 Key Pair is generated by the Next.js Node server (`actions.ts`). 
- **HttpOnly Cookies**: The exported Private Key (JWK) and the backend Access Token are stored inside heavily restricted `HttpOnly`, `Secure`, and `SameSite=Strict` cookies. The browser cannot read them.

### 2. DPoP Token Negotiation (Login)
1. The Next.js server generates a fresh ECDSA Key Pair via the Web Crypto API.
2. The Next.js server sends a request to the `.NET` backend (`/api/auth/login`), including the Public Key (JWK) inside the `DPoP` HTTP header.
3. The `.NET` `AuthController` computes the SHA-256 thumbprint of the JWK (`jkt`).
4. The `.NET` API issues a standard JWT Access Token but injects the `cnf` (confirmation) claim: `{"cnf": {"jkt": "<thumbprint>"}}`.
5. The Next.js server receives this Access Token and stores it in the `dpop_token` HttpOnly cookie.

### 3. DPoP Proof Validation (Protected Endpoints)
When the Next.js Server fetches secure data (`/weatherforecast`) from the `.NET` API, it must prove it possesses the private key associated with the token.

It generates a **DPoP Proof** (a uniquely signed JWT) for every single request. The `.NET` `DPoPMiddleware` rigorously validates this proof:
- **Signature Verification**: Verifies the signature of the DPoP proof against the embedded public key.
- **Thumbprint Binding**: Verifies that the public key inside the DPoP proof perfectly matches the thumbprint stored in the Access Token's `cnf.jkt` claim. 
- **HTTP Method Binding (`htm`)**: Ensures the proof was generated specifically for the `GET` method. Intercepted proofs cannot be used for `POST` or `DELETE`.
- **HTTP URL Binding (`htu`)**: Ensures the proof is strictly tied to the `/weatherforecast` URL. It cannot be replayed against `/admin/deleteUser`.
- **Time/Replay Protection (`iat` / `jti`)**: (Implementation ready) Prevents an intercepted valid proof from being re-used globally.

## How to Run

1. **Start the Backend (.NET 8):**
   ```bash
   cd backend
   dotnet run
   ```
   *The backend will run on `http://localhost:5083` (or the port defined in `launchSettings.json`).*

2. **Start the Frontend (Next.js):**
   ```bash
   cd frontend
   npm install
   npm run dev
   ```
   *The Next.js app will be available at `http://localhost:3000`.*

3. **Verify:**
   Open `http://localhost:3000` in your browser. Click **Login via Server**, then **Fetch Data via Server**. Observe the logs indicating the secure transmission of data entirely abstracted away from the browser console.
